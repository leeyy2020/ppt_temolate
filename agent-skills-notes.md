# Agent Skills 笔记

基于当前环境中的 Codex skills 文档整理，目标是快速说明什么是 skill、skill 适合解决什么问题、以及一个可用 skill 通常应该长什么样。

## 1. 什么是 Agent Skill

Skill 可以理解为给 agent 提供的一组模块化、可复用的“任务说明 + 资源打包”能力。

它不是单纯的提示词，而是一个目录化的能力单元，通常包含：

- 任务触发描述
- 工作流说明
- 可选脚本
- 可选参考文档
- 可选素材或模板

在当前体系里，skill 的核心价值有四类：

- 专用工作流：把某个复杂任务拆成稳定步骤
- 工具集成：指导 agent 正确使用特定格式、脚本或外部接口
- 领域知识：补足模型默认不具备的业务知识或项目约束
- 资源复用：把重复用到的模板、脚本、素材打包好

一句话概括：skill 的作用是把通用 agent 变成“在某类任务上更稳定、更省上下文、更不容易走偏”的专用 agent。

## 2. Skill 的基本结构

一个 skill 至少需要一个 `SKILL.md`，完整结构通常类似：

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
├── references/
└── assets/
```

### `SKILL.md`

这是 skill 的核心入口，包含两部分：

- YAML frontmatter
- Markdown 正文

frontmatter 至少需要：

- `name`
- `description`

这两个字段非常关键，因为它们直接影响 skill 会不会被触发。

### `agents/openai.yaml`

这是偏 UI/展示层的元数据，常见字段包括：

- `display_name`
- `short_description`
- `default_prompt`
- 图标资源

它不负责核心逻辑，但会影响 skill 在界面中的可发现性和可用性。

### `scripts/`

适合放可重复执行、对稳定性要求高的逻辑，例如：

- 文件转换
- 格式校验
- 渲染
- 批处理

### `references/`

适合放“按需加载”的说明性资料，例如：

- API 文档
- schema
- 业务规则
- 详细流程说明

### `assets/`

适合放不会直接读进上下文、但会被最终产物使用到的文件，例如：

- 模板
- 图标
- 字体
- 样例文件

## 3. Skill 的设计原则

从当前 `skill-creator` 文档看，好的 skill 大致遵循下面几条原则。

### 3.1 简洁优先

默认前提是模型已经足够聪明，不需要在 skill 里重复写常识。

应优先保留：

- 模型默认不知道的约束
- 容易做错的步骤
- 具体到任务的决策规则

不应堆积：

- 大量背景介绍
- 冗长的“教程式”说明
- 和实际执行无关的辅助文档

### 3.2 给 agent 合适的自由度

不同任务需要不同强度的约束：

- 高自由度：只有方向和启发式规则，适合开放问题
- 中自由度：给出推荐流程、伪代码或参数化脚本
- 低自由度：明确脚本和固定步骤，适合脆弱、易错任务

判断标准很简单：如果任务像“窄桥”，就多给护栏；如果任务像“开阔地”，就少设限制。

### 3.3 渐进式披露

Skill 不是把所有资料一股脑塞进上下文，而是三层加载：

1. `name + description`
2. `SKILL.md` 正文
3. `references/`、`scripts/`、`assets/` 等附加资源

这能减少上下文浪费，让 agent 只在必要时再读取细节。

### 3.4 保护验证过程

如果 skill 需要验证，最好看真实产物和真实日志，而不是只看“解释是否听起来合理”。

更可靠的验证材料包括：

- 输出文件
- 命令日志
- diff
- 渲染图
- 测试结果

## 4. 什么任务适合做成 Skill

适合做成 skill 的通常是这几类任务：

- 重复出现，流程相对固定
- 单次执行成本高，走错一步代价大
- 需要依赖专门脚本、模板或参考资料
- 任务存在明显的项目内知识壁垒

典型例子：

- PPTX 生成与版式校验
- 特定数据库或业务 schema 查询
- 固定格式文档生成
- 某个内部 API 或系统的标准化操作
- 某类代码迁移或部署流程

## 5. 什么内容不应该塞进 Skill

当前 skill 设计明确不鼓励创建很多“附属说明文档”，例如：

- `README.md`
- `INSTALLATION_GUIDE.md`
- `QUICK_REFERENCE.md`
- `CHANGELOG.md`

原因不是这些文件没价值，而是它们常常会让 skill 目录臃肿，真正关键的信息反而被淹没。

Skill 应该优先服务 agent 执行任务，而不是服务人类阅读一个完整项目文档。

## 6. 当前环境里已有的 Skills

### `slides`

用途：

- 生成和编辑 `.pptx`
- 使用 PptxGenJS
- 带渲染、越界检查、字体检测等验证流程

特点：

- 要求产出 `.pptx` 和 `.js`
- 鼓励保留可编辑对象
- 有现成脚本可做渲染和校验

这是一个很典型的“高价值 skill”例子，因为 PPT 任务：

- 细节多
- 容易版式错位
- 需要脚本辅助验证

### `skill-creator`

用途：

- 指导如何创建或更新一个 skill

特点：

- 强调元数据的重要性
- 强调 progressive disclosure
- 强调 skill 目录要保持克制

它更像“构建 skill 的方法论 skill”。

### `skill-installer`

用途：

- 从 `openai/skills` 或其他 GitHub 路径安装 skill

特点：

- 支持 curated 和 experimental 列表
- 使用 helper scripts 执行安装
- 明确提醒安装后要重启 Codex 才能生效

这说明 skill 体系本身也是可以持续扩展的。

### `openai-docs`

用途：

- 查询官方 OpenAI 文档
- 给出较新的、带来源的官方答案

特点：

- 有 `default_prompt`
- 依赖特定 MCP 文档源
- 典型适合“知识变化快、必须引用官方来源”的任务

## 7. 一个好 Skill 通常长什么样

可以把它理解成下面这个公式：

```text
好 skill = 清晰触发条件
         + 精简但有效的流程说明
         + 必要时可调用的脚本
         + 按需加载的参考资料
         + 可验证的产物标准
```

如果缺少其中任一项，常见问题会是：

- 触发条件不清：skill 不会被稳定使用
- 说明太空：agent 理解不到位
- 说明太长：上下文浪费严重
- 没有脚本：重复工作太多
- 没有验证：产物质量不稳定

## 8. 一个实用的写法模板

如果以后要自己写 skill，可以从下面这个骨架开始：

```markdown
---
name: your-skill
description: Explain what this skill does, when it should be used, and what kinds of tasks should trigger it.
---

# Your Skill

## Overview

一句话说清任务目标和核心方法。

## When To Use

- 适用场景 1
- 适用场景 2

## Workflow

1. 第一步
2. 第二步
3. 第三步

## Resources

- `scripts/...` 什么时候用
- `references/...` 什么时候读
- `assets/...` 什么时候拷贝

## Validation

- 如何确认结果是对的
- 哪些错误必须修复
```

## 9. 对 Agent Skills 的一个实用判断标准

当你在想“这件事值不值得做成 skill”时，可以问四个问题：

1. 这个任务是不是会重复出现？
2. 它是不是有清晰、稳定、可复用的流程？
3. 它是不是依赖某些专门知识、脚本或模板？
4. 没有 skill 时，agent 是否很容易犯同一类错误？

如果四个问题里至少有三个答案是“是”，那基本就值得做成一个 skill。

## 10. 总结

Skill 不是把提示词写长，而是把“触发条件、任务流程、工具资源、验证方法”打包成一个能稳定复用的能力模块。

从当前环境的几个实例看，一个成熟的 skill 往往具备三个共同特征：

- 触发条件明确
- 资源组织克制
- 结果可以验证

如果后续要继续扩展，可以在这份笔记的基础上再细分两类内容：

- 如何从零设计一个新 skill
- 如何给现有项目沉淀一套内部 skill 库
